# OpenClaw News Report Plan

## Purpose

OpenClaw belongs in Watchtower as a local/private text assistant, not in the
main VultureInv cockpit.

Watchtower should collect source material, produce concise briefs, and send
Telegram alerts. VultureInv should consume the resulting snapshots through the
backend-mediated API and remain the owner decision cockpit.

## Current Flow Check

The project flow should stay:

```text
external sources
-> Watchtower providers
-> raw/derived snapshots
-> optional local text draft by OpenClaw
-> Telegram brief/alert
-> Watchtower API
-> VultureInv backend
-> VultureInv cockpit
```

Keep this forbidden:

```text
VultureInv frontend -> OpenClaw
VultureInv frontend -> Telegram
VultureInv frontend -> external news/data sources
OpenClaw -> score/risk/position sizing
```

## OpenClaw Role

Allowed:

- summarize already collected news, filings, and macro source text,
- draft a short daily battlefield brief,
- explain why a Watchtower alert may matter,
- help Telegram answer owner commands such as `/brief` and later `/ask`,
- produce text fields with provider/model/timestamp metadata.

Not allowed:

- fetch market/news data by itself for MVP,
- calculate flow totals,
- calculate score, conviction, stop loss, max loss, or position size,
- override Watchtower snapshots,
- override VultureInv risk gates,
- auto-analyze the full universe on every schedule.

## MVP Phases

### Phase 0 - Mock Loop Stable

Goal:

- `/health`, `/status`, `/jobs/refresh/mock`, `/snapshots/*/latest` work.
- Telegram `/status`, `/refresh_mock`, `/brief` work with mock snapshots.
- VultureInv main app can show Watchtower connected/stale/offline.

Do not enable OpenClaw before this passes.

### Phase 1 - OpenClaw Health Adapter

Goal:

- Add a local AI provider adapter behind `app/ai/local_provider.py`.
- Support only a health/check method first.
- Keep live AI disabled by default.

Acceptance:

- Watchtower `/status` can show local AI configured/unconfigured/offline.
- No prompt generation yet.
- No Telegram AI output yet.

### Phase 2 - Owner-Triggered Brief Draft

Goal:

- Build a source bundle from latest stored snapshots.
- OpenClaw summarizes only that bundle.
- Telegram `/brief` can optionally append a local AI draft when explicitly enabled.

Rules:

- If OpenClaw fails, Telegram falls back to deterministic snapshot brief.
- Every AI section is labeled `LOCAL AI DRAFT`.
- Include source snapshot timestamps.
- No score/risk/position fields may be generated by OpenClaw.

### Phase 3 - News Report Snapshot

Goal:

- Store an AI-drafted report as a derived snapshot, not as product truth.
- Snapshot type can be `news_report` or `brief`.

Required metadata:

```text
snapshot_id
snapshot_type
generated_at
source_snapshot_ids[]
ai_provider: openclaw
ai_model
is_live_ai
freshness_state
items[]
warnings[]
```

VultureInv can later display the latest report through backend-mediated
Watchtower API.

### Phase 4 - Telegram Alert Policy

Goal:

- Send alerts only for high-signal conditions:
  - stale critical source,
  - job failure,
  - watched symbol catalyst/disclosure,
  - regime shift,
  - flow accumulation threshold.

OpenClaw may draft alert wording, but deterministic alert rules decide whether
an alert is sent.

## Endpoint Direction

Existing contract should stay:

```text
GET /health
GET /status
GET /snapshots/regime/latest
GET /snapshots/flow/latest
GET /snapshots/catalysts/latest
POST /jobs/refresh/mock
```

Later additions:

```text
GET /snapshots/brief/latest
POST /jobs/draft/brief
POST /telegram/test_message
```

Non-mock job endpoints must require a local token before implementation.

## Practical Next Step

Do not implement OpenClaw live calls yet.

Next implementation should stabilize Watchtower MVP first:

1. validate current scaffold,
2. make `/brief` deterministic and useful from mock snapshots,
3. expose local AI status in `/status` without calling it,
4. then add OpenClaw adapter health check.
